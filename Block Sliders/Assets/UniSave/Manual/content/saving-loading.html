Saving & Loading

When you’re done setting up your game objects, you will have to tell UniSave when you want to save the data. Saving can be done with a single line of code. An unlimited amount of save files can be created. Creating a save file with a name that already exists, will just overwrite the existing file. If you want your save file to have a custom file extension, you can include it in the name of the save file.

 

Save files come in two types:

  • Scene File
  • Object File

 

A Scene File contains the name of your scene, save data, and data about game objects that should be destroyed. Scene files can contain data about multiple scenes, allowing for MultiScene Persistency. To read more about this, please check out the MultiScene Persistency page.

An Object File doesn’t care about individual scenes, and allows you to load its saved data from any scene.

 

There are two ways to save game objects, and which one you use depends on how your game is built, and your use cases.

Saving scenes

The easiest way to save your game objects is by saving the entire scene. This what you’ll most likely use, and it can be done by calling SaveManager.SaveScene(string saveName), with saveName being the name of your save file. When saving in this manner, UniSave will look for every SaveableObject in the scene, and save its data to a save file. It will also save some information about game objects that are destroyed, so it will be able to destroy them automatically when loading the save file. This creates a Scene File.

Loading a save file is just as straightforward as saving, and can be done by calling SaveManager.Load(string saveName).

It’s important to note that you shouldn’t change the name of your scenes when using this method. When saving scene data, UniSave also stores the name of the scene as a reference, so the scene can automatically be loaded when loading the save file. If you change the name, the reference will break and your data for that particular scene won’t be loaded. This is a current UniSave limitation and it’s why it’s important to stick to a single name for a scene if you don’t want to break existing save files.

 

Saving game objects

Another method to save game objects, is by saving individual game objects (and their children) to their own save files. The type of save file that will be created here is an Object File. You usually use this in combination with the other save method, for specific use cases. This is useful in situations where you might want some saved data to be modular and loadable from any scene. For example, if you want to save configuration data for your game (controls, game settings, etc.) then saving something like configuration manager object to its own save file would make sense. You can save a single game object by calling SaveManager.SaveObject(string saveName, GameObject gameObject).

Loading the file is exactly the same as the other method, and can be done by calling SaveManager.Load(string saveName, string loadScreen).

Fields & Properties

During the saving process, UniSave looks for all the public fields and public properties on the components that were selected. Fields and properties that are private, static or read-only (constants, read-only fields and read-only properties), are ignored.

By default all the public fields and properties on a component will be saved, but if you want UniSave to ignore a member (like for efficiency reasons), you can use the UniSaveIgnoreDataMember attribute on the corresponding member:

 

 C#

    [UniSaveIgnoreDataMember] public int MyNumber

JavaScript

    @UniSaveIgnoreDataMember public var MyNumber;

 

Fields and properties can be of of any type. Whether your type is Vector3, int, or even a custom class or struct, it doesn’t matter. Lists and arrays (including jagged arrays) can be saved as well. The only exceptions are hashtables, dictionaries, and multidimensional arrays. These are currently not supported by UniSave, and will be ignored when saving.

When saving fields or properties that are references to assets (materials, textures, fonts, custom asset types, etc.) you have to make sure the asset is in a Resources folder, or else it cannot be found by UniSave. Make sure all of your assets are uniquely named when located in a Resources folder.
